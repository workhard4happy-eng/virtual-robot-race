# 4. ログデータの確認とテーブルモード

このレッスンでは、前回のマニュアル操作で保存した走行ログ（Log）がどのようなものかを確認し、そのログデータを利用して走行を再現する「テーブルモード」について学びます。

> **▶ このレッスンをまず動画で見る**: [Lesson 04 ログデータとテーブルモード（YouTube）](https://youtu.be/kwLAPdCD-jU?t=335)

**学習目標:**
- `DATA_SAVE=1`で保存されるログデータのフォルダ構成と内容を理解する
- `metadata.csv`に含まれる詳細な走行データの意味を知る
- テーブルモード（`MODE_NUM=2`）を使い、CSVファイルに基づいてロボットを走行させる方法を習得する
- ログデータを編集し、独自の走行パターンを作成する基礎を学ぶ

## 1. ログデータの確認

`03_Manual_Control`のレッスンで`DATA_SAVE=1`に設定して走行すると、`Robot1/training_data/`フォルダ内に`run_[日付]_[時刻]`という名前のフォルダが作成されます。これが1回の走行に関する全てのデータが詰まった「ログフォルダ」です。

### 1.1 ログフォルダの構成
ログフォルダの中には、主に以下のファイルとフォルダが格納されています。

- **`images/`フォルダ**
  - 走行中にロボットのカメラが捉えた全フレームの画像（JPEG形式）が保存されています。

- **`metadata.csv`**
  - その走行に関する**最も重要なデータファイル**です。走行中のロボットの全状態が、1tick（シミュレーションの1コマ）ごとに記録されています。

- **`output_video.mp4`**
  - `images/`フォルダの画像を連結して自動生成された、走行のリプレイ動画です。

- `UnityLog.txt`, `terminal_log.txt`
  - デバッグ用の詳細な動作ログです。
  - Unity側、およびPython側で処理された動作ログが保存されています。

---


### 1.2 `metadata.csv` の中身
`metadata.csv`をExcelやGoogleスプレッドシートなどの表計算ソフトで開いてみましょう。以下の **16個のカラム（列）** が左から順番に並んでいます。

#### 時系列・識別情報

| カラム名 | 内容 |
|----------|------|
| **`id`** | Tickの連番インデックス。ファイル全体を通して一意な番号です。 |
| **`session_time_ms`** | スタートシーケンス開始からの経過時間（ミリ秒）。カウントダウン中も計測されます。 |
| **`race_time_ms`** | 「GO」シグナルが発火した瞬間からの経過時間（ミリ秒）。カウントダウン中は `0` のままです。 |
| **`filename`** | `images/`フォルダ内の対応する画像ファイル名（例: `frame_000079.jpg`）。 |

> **💡 データ量の目安**: 1tick = 50ms。つまり **1,000行 ≒ 50秒分** のデータです。2周分の走行なら概ね 1,000〜2,000行になります。行数を見るだけで走行時間のおおよそが分かります。

#### ロボットの操作・状態

| カラム名 | 内容 |
|----------|------|
| **`soc`** | バッテリー残量（充電率）。`1.0` が満充電、`0.0` が空。 |
| **`drive_torque`** | 駆動輪トルクの正規化値。`+1.0` が最大前進、`-1.0` が最大後進、`0.0` が停止。 |
| **`steer_angle`** | 前輪の操舵角（**ラジアン単位**）。最大 **±0.524 rad（±30度）** の範囲。負の値（`-`）が**左**、正の値（`+`）が**右**。 |
| **`status`** | ロボットの状態を表す文字列（下表参照）。 |

**`status` の値一覧:**

| 値 | 意味 |
|----|------|
| `StartSequence` | スタートのカウントダウン中 |
| `Running` | 走行中 |
| `Lap0` | ラップ計測開始地点（学習時に使用） |
| `Lap1` / `Lap2` ... | ラップ通過後 |
| `Finish` | 規定周回を完走 |
| `FalseStart` | フライング（GO前に動いた）→ 失格 |
| `Fallen` | コース外に落下 → 退場 |
| `BatteryDepleted` | バッテリー切れ → 走行不能（障害物化） |
| `ForceEnd` | 強制終了 |

#### 位置・姿勢

> **注意**: 列の並び順は `pos_z, pos_x, yaw, pos_y` です。アルファベット順ではありません。  
> **[重要]** `steer_angle` は **ラジアン単位** ですが、`yaw` は **度単位** です。同じグラフに重ねて描く場合は単位を合わせる必要があります。

> **❓ 「なぜ `steer_angle` はラジアンで `yaw` は度なの？」**
> Unityはオイラー角を内部で「度」で管理するため、ワールド座標の向き（yaw）は度のままエクスポートされます。一方、`steer_angle` はPython側の制御モデルで三角関数（sin, cos）を使う際にラジアンが便利なため、そのままラジアンで出力しています。単位の混在はバグの温床になりやすいので、分析前に `numpy.degrees(steer_angle)` で統一することをお勧めします。

> **❓ 「分析のたびに変換コードを書くのはDRYに反するので、読み込み時に単位を自動統一するラッパークラスを作っていい？」**
> むしろ積極的にやるべきです。良いエンジニアリングの発想です。基本的な変換はこのような形です。
> ```python
> df["steer_deg"] = np.degrees(df["steer_angle"])  # ラジアン → 度
> df = df[df["race_time_ms"] > 0]                  # カウントダウン行を除外
> ```
> AIアシスタント（Gemini Code Assist、Claude Code、Codexなど）に「metadata.csvを読み込んでsteer_angleを度に変換し、race_time_ms > 0 の行だけ返す DataLoader クラスを作って」と依頼すると、再利用可能なコードを生成してくれます。
> - **Excel / Googleスプレッドシート**: `=DEGREES(steer_angle)` で度に変換できます
> - **Python (NumPy)**: `numpy.degrees(steer_angle)` または `steer_angle * (180 / numpy.pi)` で変換できます

| カラム名 | 内容 |
|----------|------|
| **`pos_z`** | Z座標 [m]。**前後方向**の位置（コース進行方向が正）。 |
| **`pos_x`** | X座標 [m]。**左右方向**の位置（右が正）。 |
| **`yaw`** | ヨー角（進行方向）[**度**]。前進方向=0、右回りが正の値。 |
| **`pos_y`** | Y座標 [m]。**上下方向**の位置（上が正）。`-0.1m` 未満でコース外落下と判定。 |

> **❓ 「座標はロボット基準のローカル座標？それともワールド座標？」**
> **ワールド座標**です。コースのスタート地点付近を原点としたUnityのワールド座標系がそのまま記録されています。Unityは**左手座標系**（Z軸が前方、X軸が右、Y軸が上）を採用しており、データもその通りです。列の並び順が `pos_z, pos_x, yaw, pos_y` とアルファベット順でないのは、レース文脈でZ（前後）が主方向、X（左右）が副方向のため優先度を反映しています。

#### エラー・衝突情報

| カラム名 | 内容 |
|----------|------|
| **`error_code`** | エラー状態を示す数値コード。通常走行時は `999`。 |
| **`collision_type`** | そのTickの衝突種別: `"wall"`（壁）、`"robot"`（他ロボット）、`"both"`（両方）、`""`（衝突なし）。 |
| **`collision_penalty`** | そのTickに適用された衝突ペナルティ率（通常は `0.0`）。バッテリーから即時減算される。 |
| **`collision_target`** | 衝突した相手の名前（例: `Robot2`、`Wall`）。衝突なしの場合は空。 |

このファイルを見ることで、「どの瞬間に、どんな操作をして、その結果ロボットがどこにいたか、衝突はあったか」を詳細に分析することができます。

## 2. テーブルモードでの走行再現

テーブルモードは、`Robot1/table_input.csv`というファイルに記述された`drive_torque`と`steer_angle`の値を上から順番に読み込み、その通りにロボットを動かすモードです。
つまり、**走行の設計図（テーブル）に基づいて、ロボットが自動で走ります。**

### `table_input.csv`の形式
このファイルは非常にシンプルで、3つのカラムしかありません。
- `time_id`: 動作の通し番号（0から始まる整数）。
- `drive_torque`: アクセル操作。
- `steer_angle`: ハンドル操作。

## 3. レッスン課題

### 課題1: ログデータから走行を再現する

1.  **事前準備**
    - `start.bat` を実行してランチャーを開き、以下のように設定します。

    | 項目 | 設定値 |
    |------|--------|
    | **Active** | `1` |
    | **R1 mode** | `table` |
    | **Data save** | `OFF` |

    設定を確認したら **START** をクリックします。

2.  **テーブルデータの作成**
    - 前回の練習で作成されたログフォルダを開き、`metadata.csv`を表計算ソフトで開きます。
    - `drive_torque`と`steer_angle`の2つの列のデータを、ゴールするまでの行（例: 1000行目までなど）選択してコピーします。

3.  **`table_input.csv`の編集**
    - `Robot1/table_input.csv`をテキストエディタや表計算ソフトで開きます。
    - **A列(`time_id`)以外の、`drive_torque`, `steer_angle`の古いデータを全て削除**します。
    - 先ほどコピーした`drive_torque`, `steer_angle`のデータをB列、C列に貼り付けます。
    - **A列(`time_id`)を、`0`から始まる連番で埋め直します。** Excelなどを使う場合は、`0`、`1`と入力してフィルハンドルをドラッグすると簡単です。
    > **💡 A列を埋めないとどうなる？**: `time_id` が空欄や重複していると、プログラムが「何行目のデータを読めばいいか」を判断できず、正常に動作しません。面倒でも必ず連番で埋め直してください。
    - 編集が終わったら、ファイルをCSV形式で保存します。

4.  **走行の確認**
    - `start.bat` を実行します。
    - ロボットが、先ほどコピーしたログ通りに自動で走行を始めることを確認しましょう。自分のキーボード操作が再現される様子は、少し不思議な感覚かもしれません。

> **考えてみよう: 完全に同じ動きになりましたか？**
>
> きれいに周回できた人は少ないのではないでしょうか。これは**「遅れ（レイテンシ）」**が原因です。
>
> 皆さんがログとして使ったデータは、ロボットが**実際に受信した**操作値です。そして今回のテーブルモードでは、そのデータを上から順番に **20fps（50ms周期）** で送信しました。ロボット側はそれを受け取って動きます。
>
> しかし、実際にはいくつかの「ズレ」が生じます。
> - **通信の遅れ**: 送信が必ず成功するとは限らず、パケットの到着タイミングにばらつきがある
> - **機械的な遅れ**: トルクを指示してから実際にタイヤが回転するまでには、わずかな時間差がある
>
> **なぜ通信が必要なの？** Unityのシミュレーターとあなたのアルゴリズム（Pythonプログラム）は、**別々のアプリとしてWebSocket（ネットワーク通信）を介してつながっています**。ゲームのTASのように「同じ入力 = 完全に同じ結果」とはならないのは、この通信遅延が原因です。**これは仕様であり、あなたのコードやPCの問題ではありません。**
>
> これは現実の世界でも同じです。「決められた指令通りにただ動く」だけでは、外からの影響（外乱）には対応できません。このような制御方式を**シーケンス制御**と呼びます。
>
> 思い通りに動かすためには、余裕を持った指令値の工夫、またはこの後学ぶ**フィードバック制御**が必要になります。

> **❓ 「Unityはコマンドをタイムスタンプ通りに実行するのではなく、届いた瞬間に即実行しているの？」**
> その通りです。Unityは受信したコマンドをバッファリングせず、届いたフレームで即座に適用します。タイムスタンプによる遅延補償は行っていないため、通信のジッタ（揺らぎ）がそのまま制御のズレとして現れます。

> **❓ 「なぜ50ms（20fps）という制御周期にしたの？もっと速くできないの？」**
> 複数の要因のバランスです。①WebSocketのラウンドトリップ遅延（ローカル通信でも数ms）、②Python側のAI推論時間（CPU環境でも50ms以内に収まる設計）、③Unityのレンダリング負荷。数msの超高速周期を実現するにはWebSocketではなく共有メモリやUDPが必要になりますが、導入の複雑さより**学習しやすさ**を優先してWebSocket+50msを採用しています。

> **❓ 「テーブルモードってオープンループ制御（シーケンス制御）ですよね？次はPID制御が来る？」**
> その通り、テーブルモードはセンサーを一切参照しない**オープンループ制御**です。初期位置の微小なズレやジッタが積分されて誤差が拡大するのは必然です。次のLesson 05では、カメラ画像からレーンのズレを検出してステアリングを修正する**閉ループ制御**を実装します。PID制御でいう比例項（P制御）に相当するアプローチを体験できます。

> **❓ 「Unity側でタイムスタンプ付きバッファリングキューを実装して遅延補償することは可能？」**
> 技術的には可能です。UnityのC#側でタイムスタンプ付きコマンドキューを実装し、Python側が各コマンドに送信時刻を付与すれば、より精密な再現が実現できます。ただし現実的な制約があります。①ローカルホスト通信のジッタは通常1〜5ms程度で50ms周期内への影響は限定的、②Unityの物理エンジン更新レート（デフォルト50fps）自体がボトルネックになる、③実装の複雑さが大幅に増す。「完全再現」より**外乱に強いフィードバック制御を設計する**ことの方が、実用的かつ本質的なアプローチです。

> **❓ 「UDPや共有メモリで制御周期を10msまで引き上げることは可能？」**
> 通信層だけなら可能です。Pythonの`asyncio + socket`（UDP）や`multiprocessing.shared_memory`で数ms以下の遅延を実現でき、UnityのC#側も`System.IO.MemoryMappedFiles`で対応できます。ただし、①Unityの`Fixed Timestep`（デフォルト50fps）を変えなければシミュレーター側の更新が追いつかない、②WebSocketが提供する再送・順序保証をUDPでは自前実装する必要がある、という課題があります。MPCなど高度なアルゴリズムの研究テーマとして挑戦する価値はあります。

### 課題2: 走行データを編集してみる（応用）

- `table_input.csv`の特定区間の`steer_angle`を全て`0`に書き換えて保存し、ロボットがどのように動くか観察してみましょう（直線的に走るはずです）。

### 課題3: `metadata.csv` を可視化してみよう

ログデータは数字の羅列ですが、**グラフにすることで走行の様子が「見える」**ようになります。
ExcelやGoogleスプレッドシート、あるいはPython（`pandas` + `matplotlib`）など、使いやすいツールで試してみましょう。

> **💡 Pythonでのグラフ化、コードの書き方が分からない場合**: AIアシスタント（Gemini Code Assist、Claude Code、Codexなど）に以下のように頼んでみましょう。
> ```
> Robot1/training_data/ 以下にある metadata.csv を読み込んで、
> race_time_ms を横軸、drive_torque と steer_angle を縦軸にした
> 折れ線グラフを描くPythonコードを書いてください。
> ```
> コードをそのまま貼り付けて実行するだけで動くコードを生成してくれます。

#### 課題3-1: 走行ルートを描く（散布図）

`pos_z`（横軸）と `pos_x`（縦軸）を散布図でプロットしてみましょう。

> **ヒント**: X軸に `pos_z`、Y軸に `pos_x` を設定し、データ点を線でつなぐと走行軌跡になります。

- コースのどんな形が見えますか？
- スタート地点とゴール地点はどのあたりですか？
- 周回ごとに軌跡がずれている箇所はありますか？

#### 課題3-2: 操作の時系列を見る（折れ線グラフ）

横軸を `race_time_ms`、縦軸を `drive_torque` と `steer_angle` として折れ線グラフを描いてみましょう（2本の線を重ねて描くと比較しやすいです）。

> **ヒント**: `race_time_ms` が `0` の行はカウントダウン中です。値が `0` より大きい部分だけを使うと、実際の走行区間が分かりやすくなります。

- アクセルを踏み込んでいる区間と、緩めている区間はどこですか？
- ハンドルを大きく切っている瞬間は、`drive_torque` とどんな関係がありますか？
- カーブと直線で、操作のパターンに違いは見えますか？

---

#### 課題3-3: 位置と向きの時系列を見る（折れ線グラフ）

横軸を `race_time_ms` として、`pos_z`・`pos_x`・`yaw` の3本を時系列グラフで描いてみましょう。

- `pos_z` と `pos_x` が大きく変化するタイミングは一致していますか？それとも交互に変化しますか？（それは何を意味しますか？）
- `yaw` が急激に変化している区間は、散布図（課題3-1）のどの部分に対応していますか？
- `yaw` の変化と、課題3-2の `steer_angle` の変化を見比べてみましょう。どんな関係が見えますか？

#### 課題3-4: さらに挑戦したい方へ（オプション）

> **💡 3D走行軌跡**: `mpl_toolkits.mplot3d` を使って `pos_x`・`pos_z`・`pos_y` を3次元プロットすると、コースの立体構造（坂・段差）が見えます。

> **💡 操作ヒートマップ**: X軸を `steer_angle`、Y軸を `drive_torque` にした2次元ヒストグラム（`plt.hist2d`）を作ると、自分の運転のクセ（ブレーキ習慣・ステアリングの偏り）が一目で分かります。AIアシスタントに「steer_angleとdrive_torqueの2Dヒストグラムを描いて」と頼んでみましょう。
