### 2.1 推論の本体コード

ファイルの中心にあるのは、たった数行のコードです：

```python
# 入力テンソルの準備
image_tensor = _transform(pil_img).unsqueeze(0).to(_device)  # [1, 3, 224, 224]
soc_tensor = torch.tensor([[soc]], dtype=torch.float32).to(_device)  # [1, 1]

# 推論の実行
with torch.no_grad():
    output = _model(image_tensor, soc_tensor)
    raw_drive = output[0, 0].item()   # drive_torque
    raw_steer = output[0, 1].item()   # steer_angle
```

**これがすべてです。** 2つのデータを入力し、2つのデータが返ってくる。これがMIMOです。

---

### 2.2 モデルの構造（`model.py`）

ニューラルネットワーク（`DrivingNetwork`）の構造はシンプルです：

```
入力①: RGB画像 224×224      → CNN（畳み込み4段 + GlobalAvgPool）→ 256次元ベクトル
                                                                      ↓ 結合（concat）
入力②: SOC（浮動小数点1個） ─────────────────────────────────────→ 257次元ベクトル
                                                                      ↓ MLP（全結合）
出力: [drive_torque, steer_angle]（2次元）
```

> **補足**: CNN（Convolutional Neural Network）は画像から特徴を抽出します。
> 「コースが右にカーブしている」「白線が左に寄っている」などの視覚情報を数値化します。

> **Q: 速度計がないのに単一フレームだけで大丈夫？**
> 速度をそのまま測る手段はありません。ただし、これは意図的なトレードオフです。RNNや複数フレームのスタックを使えば速度情報を推定しやすくなりますが、学習データ量が増え、実装が複雑になり、50ms以内の推論制約も厳しくなります。単一フレームCNNを選んだのは「シンプルさと速度の両立」が理由です。十分な量のデータがあれば、コース上の白線の相対位置からその状況に適したトルク・ステアを学習できます。

### 2.3 Hybridモード vs Pure E2Eモード

このシステムには2つのモードがあります（`ai_control_strategy.py` で設定）：

| モード | スタート検出 | 走行制御 | 特徴 |
|--------|------------|---------|------|
| **hybrid** (デフォルト) | ルールベース | AI | スタート合図の検出は確実なルールで処理 |
| **pure_e2e** | AI | AI | 完全にAIに任せる（スタート合図も学習対象） |

> ルールで確実に処理できることをわざわざAIに学習させる必要はありません。
> 「決まり切った処理はルール、判断が難しい処理はAI」が現実的なアプローチです。

**`ai_control_strategy.py` でできる高度なカスタマイズ（上級者向け）：**

ファイルを開くと、モード切り替え以外にも多くのパラメータが設定できます。

| パラメータ | 内容 |
|-----------|------|
| `START_BOOST_FRAMES` | スタート直後の加速時間（フレーム数） |
| `MIN_DRIVE_TORQUE` | スタートブースト中の最低トルク |
| `STEER_SMOOTHING_ALPHA` | ステアリングの平滑化係数（0〜1） |
| `CORNER_MIN_DRIVE_TORQUE` | カーブ中のトルク上限 |

AIが不安定に走る場合は、これらの値を調整することで改善できることがあります。

## 3. AIモデルを学習させる

### 3.1 「模倣学習」とは何か？

このシステムが使っている学習手法は**模倣学習（Imitation Learning）**です。

1. 人間（あなた）がキーボードで走る
2. そのときの「画像・SOC → アクセル・ハンドル」というデータを記録
3. AIは「人間ならこの画像でこう動く」というパターンを学習する

つまり、**AIは「あなたの運転を真似る」** ように学習します。
うまく走れるAIを作りたければ、うまく走ったデータを与えることが重要です。

> **重要**: この学習はレース中にリアルタイムで行われるのではありません。
> 走行データを収集 → **オフラインで学習** → 学習済みモデルで走行、という流れです。

---

### 3.3 ローカルでの学習手順

最低でも3回分のデータがあると学習の効果が出やすいです。

**Step 2: 学習を実行する**

`Robot1/` ディレクトリに移動してから学習を実行します：

```bash

学習が始まると、エポックごとにlossが表示されます：

```

Epoch   2/100 | Train: 0.038441 | Val: 0.041023 | LR: 1.00e-04 | 12.1s ✓ NEW BEST
...
⏹️  Early stopping triggered at epoch 47
Best validation loss: 0.012345
Model saved to: experiments/iteration_[timestamp]/model.pth
```

**Step 3: モデルを配置する**

生成されたモデルを `models/` フォルダにコピーします。まず、作成されたフォルダ名を確認します。

```bash
ls experiments/
```

`iteration_20260502_143022` のようなフォルダが表示されます。その名前を使って以下のコマンドを実行します（`iteration_[日付時刻]` の部分は実際のフォルダ名に置き換えてください）。

> **PowerShell の `copy` について**: PowerShell では `cp` も `copy` も使えます。パスに `[]` が含まれると誤動作することがあるため、実際のフォルダ名（数字だけ）に置き換えてから実行してください。

> **ポイント**: `model.pth` (最終エポックのモデル) よりも `model_best.pth` (検証ロスが最小だったモデル) を使う方が、性能が良い傾向にあります。

### 3.3.5 結果を読み解く：なぜうまくいった？なぜうまくいかなかった？

走らせた結果を、以下の観点で分析してみましょう。

**うまくいかなかった場合によくある原因：**

| 症状 | よくある原因 | 対策 |
|------|------------|------|
| スタートしてすぐコースアウト | データが少ない（3回未満）/ キーボード操作が荒かった | データを増やす・丁寧に走る |
| 最初のカーブで失敗する | そのカーブをうまく走ったデータが少ない | 苦手な場所を重点的に走行して追加 |
| ふらふらして直進できない | ステアリング操作が細かく振れていた | できるだけ滑らかに操作して再収集 |
| スタートしない（永久に止まる） | スタート合図のシーン未学習（pure_e2eモードの場合） | hybridモードに戻す |
| 2周目で崩れる | 1周目のデータばかりで2周目が少ない | 2周完走したデータを複数用意する |

**うまくいった場合も改善の余地があります：**

- タイムを計測しましょう。[レッスン05のルールベース](05_Rule_Based_Control.md)と比べてどちらが速いですか？
- SOCはどのくらい残っていますか？バッテリーを無駄遣いしていませんか？
- `Robot1/training_data/` の `metadata.csv` をグラフ化して、どこでエネルギーを使っているか確認しましょう（[レッスン04](04_Log_and_Table_Mode.md) の手法が使えます）
- `analyze.py` を使うと、トルク・ステアリングの分布をグラフで確認できます（`Robot1/` ディレクトリで実行）：

```bash
# 全runのトルク・ステアリング分布グラフを表示
python ai_training/analyze.py data

# テキストで全runのサマリーを確認
python ai_training/analyze.py summary
```

**数値の読み方：**

| 列 | 意味 | 良い傾向 | 注意サイン |
|----|------|---------|-----------|
| `race_time_sec` | ラップタイム | 速い・run間でばらつき小 | あるrunだけ極端に遅い |
| `torque_std` | アクセルのブレ | 低い（滑らかな運転） | 高い（ガクガクした操作） |
| `steer_mean` | 平均ステア量 | 0に近い（左右バランス） | 大きく偏っている |
| `steer_std` | ステアのブレ | 低い（安定した操作） | 高い（ふらついた運転） |

たとえば `steer_std` が他の run より突出して高い run は「ふらついた走行データ」の可能性があります。AIがそのクセを学習してしまうかもしれません。

**AIアシスタントに相談して次の一手を考える：**

```
python ai_training/analyze.py summary を実行したら以下のデータが出ました：

[Training Data Report]
run_name             final_status  race_time_sec  torque_mean  torque_std  steer_mean  steer_std
run_20260502_063425  Finish        26.394         0.644        0.425       0.298       0.276
run_20260502_063547  Finish        22.315         0.757        0.374       0.319       0.275
run_20260502_063633  Finish        27.254         0.589        0.421       0.180       0.388

←（自分の実際の数値に置き換えてください）

このデータを見て：
1. どのrunの品質が良く、どのrunが学習に向いていないか教えてください
2. このデータで学習すると、AIはどんな走り方をしそうか予測してください
3. 次の一手として何をすればいいか提案してください
```

**次の一手：データを増やして再学習**

コースアウトした場合でも「悔しがらず、データを増やす」のが正解です。

```
改善サイクル:
  現在のAIを観察 → 弱点を特定 → その場面のデータを追加収集 → 再学習 → また走らせる
```

特に苦手な場所（コースアウトしやすいカーブなど）を重点的にキーボード走行で走り、データを追加してから `python ai_training/train.py --data training_data` を再実行してみましょう。

## 4. Google Colabで学習する
> **▶ 動画チャプター**: [Google Colabで学習する](https://youtu.be/kwLAPdCD-jU?t=691)

### 4.1 なぜColabを使うのか？

ローカルPCにGPUがない場合、学習に非常に時間がかかります（数十分〜数時間）。
Google ColabのGPUを使うと、**同じデータでも数分〜10分程度**で学習が完了します。

> **体感してほしいこと**: ローカルで1時間かかっていた学習が、Colabでは5分で終わることがあります。これが「GPUの力」です。AIモデルの開発では「どれだけ速く試行錯誤できるか」が重要で、学習が速いほど改善サイクルを多く回せます。クラウドGPUを使うことの価値を、数字で体感してください。

これを**「学習コスト」**と呼びます。GPUの性能差と、クラウドに課金する意味を体感してください。

## 5. バッチ学習とリアルタイム学習の違い — あなたの現在地を知る

ここまでで、あなたは**自分のデータからAIモデルを作り、実際に走らせる**という体験をしました。

ここで一度立ち止まって、「自分が今どんな学習をしているのか」「AIの世界ではどんな手法が存在するのか」を整理しておきましょう。これを知ることで、「なぜうまく走れないのか」「次に試すべきことは何か」が見えてきます。

---

**あなたが今やっているのは「模倣学習（バッチ学習）」です**

```
バッチ学習（模倣学習）:  ← 今ここ
  キーボード走行 → データ収集 → オフラインで学習 → モデル配置 → 走行テスト → 繰り返し
```

「バッチ学習」というのは「まとめてデータを集めてから学習する」という意味です。走行中にリアルタイムで学習しているわけではありません。

---

**その先にある「強化学習（リアルタイム学習）」との違い**

```
リアルタイム強化学習:
  走行しながら → 報酬を計算 → その場でモデルを更新 → 走り続ける
```

ゲームAIや自動運転の研究でよく登場する「強化学習」はこちらです。走りながら自分で「良い動き・悪い動き」を学習し、どんどん上手くなっていくイメージです。

**airaがバッチ学習を採用しているのは、意図的な設計です**

airaは「まず基礎をしっかり体で覚える」ことを最優先に設計されています。

リアルタイム強化学習は非常に強力な手法ですが、報酬の設計・センシングの仕組み・学習の安定化など、習得すべき概念が一気に増えます。いきなりそこから始めると、「何が起きているのかわからないまま動く」という状態になりがちです。

**airaでは、まず「データを集めて学習する」という最も基本的なサイクルを体験します。**
この感覚——「どんなデータが必要か」「なぜうまく走れないのか」「何を改善すべきか」を自分で考える力——が、将来ヒューマノイドロボットや自律移動システムのような、より複雑なAI開発に取り組むときの土台になります。

> **今できる「次の一手」は Section 6 で紹介するDAggerです。** バッチ学習を賢く反復する手法で、強化学習の考え方の入り口にもなっています。

## 6. より高度な学習：DAggerと報酬設計

### 6.1 模倣学習の限界

最初に学習したモデルには弱点があります。**人間が走ったことのない場面**に遭遇すると、正しく対処できません。

例えば：
- 少しコースをはみ出した場面 → 人間のデータにないので、回復方法を知らない
- ラップを重ねるごとに少しずつズレていく → 最初だけうまくても崩れる

これを「**分布シフト（Distribution Shift）**」と呼びます。

---

### 6.2 DAgger：反復的なデータ収集で改善する

この問題を解決するヒントになる手法が **DAgger（Dataset Aggregation）**です。

> **用語の整理：Imitation Learning と Behavioral Cloning の関係**
> - **Imitation Learning（模倣学習）**：人間の操作データからAIを学習させる手法の総称
> - **Behavioral Cloning（bc）**：その中で最もシンプルな実装。人間の操作を教師あり学習で直接コピーする
> - **DAgger**：データ収集方法を工夫した Imitation Learning の亜種
>
> つまり `train.py --mode bc` で行っていることは「Imitation Learning のうち、Behavioral Cloning という手法を使っている」ということです。

**具体的な例で理解する:**

```
【最初】キーボードで3回走ってモデルを作った状態
  training_data/
    run_20260502_063425/   ← キーボード走行①
    run_20260502_063547/   ← キーボード走行②
    run_20260502_063633/   ← キーボード走行③

  → python ai_training/train.py --data training_data
  → model_best.pth を models/model.pth にコピー

【DAgger 1サイクル目】
  AIモードで走らせる（DATA_SAVE=ON）→ run_04 が生成される
  training_data/
    run_20260502_063425/   ← キーボード走行①
    run_20260502_063547/   ← キーボード走行②
    run_20260502_063633/   ← キーボード走行③
    run_20260502_091012/   ← AIが走った④ ★新しく追加

  → python ai_training/train.py --data training_data
     ↑ 01〜04 すべてをまとめて使って再学習する（削除しない）
  → 新しい model_best.pth を models/model.pth にコピー

【DAgger 2サイクル目】
  また走らせる → run_05 が追加される
  → 01〜05 すべてで再学習 → さらに改善
```

> **Q: AIが走ったデータ（run_04）だけで学習するの？**
> **A: いいえ、すべてまとめて使います。** キーボードデータ（01〜03）を残したまま、AIデータ（04）を「追加」して学習します。これがDAggerの「Aggregation（集積）」という考え方です。人間のデータを捨てると、AIが学べる手本がなくなってしまいます。

走らせるたびに「AIが苦手な場面のデータ」が `training_data/` に蓄積されていくので、繰り返すほど性能が上がっていきます。これがDAgger（Dataset Aggregation＝データの集積）の考え方です。

> **💡 Colabで学習する場合**: ローカルでAIを走らせてデータを収集 → `sync_to_gdrive.py --sync-new` でアップロード → Colabで学習 → model.pth をダウンロードしてローカルに配置、という流れで同じサイクルが回せます。

> **Q: DAggerを何サイクルも回すとキーボードデータが「薄まって」しまいませんか？**
>
> 心配な点は2つに分けて考えられます。
>
> **① AIデータの品質が低い場合** → `run_scorer.py` のフィルタが守ってくれます。落下した run はスコアが -500 付近になり `min_score=0.0` で自動除外されます。`--top-percent 30` のように絞り込みを厳しくすれば、高スコアの run（主にキーボードデータ）を優先的に使えます。
>
> **② AIデータが増えてキーボードデータの割合が下がる** → これは DAgger の狙い通りです。AI が上手くなるほど AI のデータ自体が「良いお手本」に近づくため、必ずしも問題にはなりません。もし AI の品質がキーボードより明らかに劣る状態が続く場合は、`--top-percent` を絞って高品質な run だけを学習に使いましょう。

> **⚠️ もっと詳しく知りたい方へ：** 論文で定義される「本物のDAgger」では、AIが走った状態に対して**人間が正解の操作を後からラベリングし直します**。このシステムでは人手のラベリングなしにAIの出力をそのまま学習データとして追加し、`run_scorer.py`でスコアの低いrun（落下・FalseStart等）を除外する**簡略版**を採用しています。この違いにより、AIが未経験の状況で誤った操作をした場合、その誤りも学習データに含まれる可能性があります。完走できたrunだけを使うようスコアフィルタを調整することで軽減できます。

### 6.3 報酬設計：データの「質」を数値化する
> **▶ 動画チャプター**: [報酬設計とスコアリング](https://youtu.be/kwLAPdCD-jU?t=691)

DAggerで集めたデータをすべて均等に使うのではなく、
**「良い走り」のデータを重く、「悪い走り」のデータを軽く（または除外）** 使う仕組みがあります。

これを担うのが `rl_reward.py` と `run_scorer.py` です。

| ファイル | 役割 |
|---------|------|
| `rl_reward.py` | 走行の「良さ」の評価基準（報酬の重み）を定義 |
| `run_scorer.py` | `metadata.csv` を分析して各runにスコアをつける |
| `train.py` | スコアに基づいてデータを選別・重み付けして学習に使う |

> **注意**: `rl_reward.py` は**リアルタイムでモデルを更新するためではありません**。
> レース終了後に `metadata.csv` を読んで走行品質をスコアリングするための「採点基準」です。
> 現バージョンでは、位置情報（pos_x, pos_z）はリアルタイムに取得できないため、
> 走行中にリアルタイムで強化学習を行うことは実用的ではありません。

> **💡 Colabで学習する場合：**
> Colabのノートブックは `run_scorer.py` を使わず、アップロードされた run_ フォルダをすべて学習に使います。
> スコアで絞り込みたい場合は、**ローカルで選別してからアップロード**するのが現実的です。
>
> ```bash
> # ローカルでスコアを確認し、良いrunを把握する
> python ai_training/run_scorer.py --top-percent 30
>
> # 選んだrunだけを Google Drive に送る
> python scripts/sync_to_gdrive.py --sync-new
> ```
>
> 「Google Drive にアップロードしたフォルダ＝Colabで学習に使われるもの」なので、
> 事前に選別すれば Colab でも同じ効果が得られます。

> **よくある誤解**: 「タイムのスコアを厳しくすれば、AIが勝手に速く走るようになるの？」
> → **なりません。** スコアは「どのデータを学習に使うかの選別基準」です。AIの強化学習（試行錯誤しながら自動で上手くなる）とは異なります。このシステムでは「**良い走りをした人間のデータを選別して学習させる**」という模倣学習が基本です。AIが速くなるには、速く走れた人間のデータが必要です。

### 6.4 報酬の中身

`rl_reward.py` と `run_scorer.py` には現在以下の評価基準が実装されています：

| 評価項目 | 内容 |
|---------|------|
| 完走ボーナス | 2周完走（+1000）、1周完走（+400）|
| タイム | 速いほど高スコア（基準120秒、1秒ごとに-2）|
| SOC効率 | バッテリーを残して完走するほど高スコア |
| 操作の滑らかさ | ハンドルのジャークが少ないほど高スコア |
| ペナルティ | 落下（-500）、強制終了（-100）|

この「採点基準」を書き換えることで、どんな走りを「良い走り」と定義するかを変えられます。

**スコアの使われ方（train.pyの実際の動作）：**

| フィルタ方式 | 内容 | デフォルト |
|------------|------|---------|
| `filter_runs_by_score(min_score=0.0)` | スコアが0以上のrunのみ使用 | 落下（-500）・強制終了（-100）は自動除外 |
| `get_top_runs(top_percent=50.0)` | スコア上位N%のみ使用 | 50%（全runの上位半分） |

スコアを確認したい場合は以下のコマンドで可視化できます（`Robot1/` ディレクトリで実行）：

```bash
python ai_training/run_scorer.py
python ai_training/run_scorer.py --top-percent 30
```

#### 上級者向け：AWR（Advantage-Weighted Regression）モード

`train.py` には、スコアをそのまま使う通常モード（`bc`）に加えて、より高度な学習モードが実装されています：

| モード | 正式名称 | 内容 |
|-------|---------|------|
| `bc` (デフォルト) | Behavioral Cloning（行動クローニング） | 模倣学習の中で最もシンプルな実装。全フレームを均等に学習 |
| `rw` | Reward-Weighted BC | runスコアを各フレームの損失関数の重みとして使用 |
| `awr` | Advantage-Weighted Regression | `ValueNetwork` でV(s)を学習し、A(s,a)=Q(s,a)−V(s) で重み付け |

**実行コマンド例（`Robot1/` ディレクトリで実行）：**

```bash
# ① bc（デフォルト）― まずはこれ
python ai_training/train.py --data training_data

# ② rw ― スコアの高い run を重視して学習
python ai_training/train.py --data training_data --mode rw

# ③ rw ＋ 上位50%に絞る
python ai_training/train.py --data training_data --mode rw --top-percent 50

# ④ awr ― ValueNetwork で価値関数も同時に学習
python ai_training/train.py --data training_data --mode awr

# ⑤ awr ＋ 上位50%に絞る
python ai_training/train.py --data training_data --mode awr --top-percent 50

# ⑥ 既存モデルをベースにファインチューニング（rw の場合）
python ai_training/train.py --data training_data --mode rw --finetune models/model.pth
```

**典型的な進め方：**

```
Step 1. bc でまず動くモデルを作る
  → python ai_training/train.py --data training_data

Step 2. データが 10 run 以上になったら rw を試す
  → python ai_training/train.py --data training_data --mode rw --top-percent 50

Step 3. rw でさらに改善が止まったら awr を試す
  → python ai_training/train.py --data training_data --mode awr --top-percent 50
```

> **💡 バッチサイズ・学習率をカスタマイズしたい場合（上級者向け）**
>
> `train.py` には `--batch-size` / `--lr` / `--epochs` オプションがあります：
> ```bash
> python ai_training/train.py --data training_data --batch-size 64 --lr 0.0005 --epochs 50
> ```
> Colab（T4 GPU、VRAM 16GB）でバッチサイズを大きくしたい場合は、ノートブックの Cell 6 にある `BATCH_SIZE = 32` を直接書き換えてください（128〜256 まで増やせます）。ただしこのモデルサイズ（約450万パラメータ）ではバッチサイズの恩恵は限定的です。まずはデフォルトのまま動かすことを優先してください。

AWRでは「現在の状態で平均的に期待できる報酬より、その操作で実際に得られた報酬が高かった場面」を重点的に学習します。`model.py` の `ValueNetwork` クラスがこの価値関数の実装です。

`run_scorer.py` で計算したrunスコアが「報酬（Q値の近似）」として使われます。つまりスコアリングと AWR は密接に連動しています。

> **注意**: AWRモードは通常の `bc` モードより多くのデータが必要で、学習も複雑になります。まず `bc` で動くモデルを作ってから試すことをお勧めします。

AIアシスタントと議論してみましょう。状況に合わせて選んでください。
