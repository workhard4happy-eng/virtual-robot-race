この姿勢が、ブラックボックスを使いこなすのではなく、**AIと真に共同開発するエンジニア**への第一歩です。

## 1. このシステムは何をしているのか？（MIMO制御）

### 1.1 入力と出力

このロボットの制御システムは、**MIMO（Multiple Input, Multiple Output）**と呼ばれる構造を持っています。

| 種別 | データ | 詳細 |
|------|--------|------|
| **入力① 画像** | RGB画像 | 224×224ピクセルのJPEG（ロボット搭載カメラ映像） |
| **入力② バッテリー** | SOC | 浮動小数点値（例: `0.85`）。バッテリー残量を0.0〜1.0で表す |
| **出力① アクセル** | `drive_torque` | -1.0〜+1.0の正規化トルク |
| **出力② ハンドル** | `steer_angle` | ±0.524 rad（±30度）の操舵角 |

たった**2種類の入力**から、**2種類の出力**を決定するのが制御の仕事です。

> **🎯 これは意図的な制約です。**
> 速度計はありません。GPSもありません。他のロボットの位置を教えてくれるセンサーもありません。
> 与えられるのは「景色（カメラ映像）」と「残りエネルギー」だけ。
> 人間のドライバーが目だけを頼りに速度とハンドルを判断するように、あなたのアルゴリズムも**視覚だけで走ります。**
> これがこのシステムの挑戦であり、面白さです。

### 1.2 何を達成するか？

**目標**: 2周を最短タイムで完走する。

**制約条件:**

| 制約 | 内容 | 違反したら |
|------|------|-----------|
| フライングしない | GOシグナル前に動かない | 即失格（FalseStart） |
| 落ちない | コース外に落下しない | 退場（Fallen） |
| バッテリーを使い切らない | SOCが0になると走行不能 | 障害物化（BatteryDepleted） |

> **補足**: 壁や他ロボットへの衝突はSOCが減少するペナルティですが、即失格にはなりません。ただし積み重なれば致命的になります。

**SOCの消費ルール:**

SOCは2つの要因で減少します。

| 要因 | 詳細 |
|------|------|
| **走行消費** | `drive_torque` の絶対値に比例して減少。アクセルを踏むほど消費が大きい |
| **衝突ペナルティ** | 壁や他ロボットへの衝突のたびに追加でSOCが減少 |

つまり「速く走る＝バッテリーを多く使う」というトレードオフがあります。スピードと省エネのバランスが勝敗を分ける設計要素の一つです。

### 1.3 タイミング

制御ループは **20fps（50ms周期）** で動作します。50msごとに画像とSOCを取得し、出力を決定して送信します。

> **⚠️ 処理が50msを超えたら？**
> アルゴリズムの処理が50msを超えても、システムはエラーで止まりません。その場合は**前フレームのコマンド（アクセル・ハンドル）がそのまま維持**されます。つまり重い処理を書くと、ロボットが古い指示で動き続け、カーブで壁に突っ込む可能性があります。AIアシスタントに複雑なコードを書かせた場合は、処理速度も確認しましょう。

### 2.4 なぜこの構造なのか

#### `rule_based_input.py` の役割

`main.py` はこのファイルの2つの関数だけを呼び出します。

```python
soc = data_manager.get_latest_soc(robot_id)        # SOC: float 0.0〜1.0
rgb_path = data_manager.get_latest_rgb_path(robot_id)  # 画像パス
pil_img = Image.open(rgb_path).convert("RGB")       # PIL画像として読み込み
```

```python
def get_latest_command():
    return {
        "type": "control",
        "robot_id": robot_id,
        "driveTorque": round(float(driveTorque), 3),
        "steerAngle": round(float(steerAngle), 3),
    }
```

`update()` が50msごとに呼ばれてアルゴリズムを実行し、結果をグローバル変数に書き込みます。`get_latest_command()` はその値をWebSocket送信用に返します。**この2つの関数名を変えると動かなくなります。**

#### `data_interactive/` の役割

Unityが50msごとに画像とSOC値をここに書き込みます。`rule_based_input.py` はここから読み取るだけです。直接編集するとシステムが誤動作するため、触らないでください。

### 2.5 サンプルアルゴリズムの構成

サンプルとして以下のファイルが用意されています。AIアシスタントに説明させながら読み進めるのが効率的です。

```
rule_based_algorithms/
├── sliding_windows.py          白線検出（スライディングウィンドウ法）
├── driver_model.py             操舵・速度の決定ロジック
├── perception_Startsignal.py   スタート合図の検出（カメラ映像の信号機色を画像処理で判定）
├── perception_Lane.py          レーン認識ユーティリティ
├── perception_trackposition.py トラック上の位置推定
├── status_Robot.py             ロボットの状態管理
├── Linetrace_white.py          白線トレースの実装例
└── debug_utils.py              デバッグ画像の生成（Robot1/debug/output/ に自動保存）
```

**推奨する読み順:**

全ファイルを一度に読もうとすると迷子になります。AIアシスタントに説明させる際も、以下の順番で進めるのが効果的です。

1. `rule_based_input.py` — 司令塔。全体の流れを把握する
2. `sliding_windows.py` — 画像から何を読み取っているか理解する
3. `driver_model.py` — 読み取った情報をどうアクセル・ハンドルに変換するか理解する

この3つを理解すれば、残りのファイルは「補助部品」として必要に応じて読めば十分です。

ここからが本番です。AIアシスタントに以下のプロンプトを順番に投げかけて、システムを理解しましょう。

**Step 1: カレントディレクトリの確認**
```
カレントディレクトリにアクセスできますか？
```

**Step 2: システム全体の把握**
```
ディレクトリ内の .md ファイル、.py ファイルを読み込み、
いったいどんなシステムなのか、どんなアプリかを理解してください。
```

**Step 3: ルールベースアルゴリズムの分析**
```
"rule_based_algorithms" フォルダの中身を理解し、
どんな制御がサンプルとして実装されているか説明してください。
```

> **ポイント**: AIアシスタントはコードを読んで説明してくれます。コードが読めなくても、AIが「何をしているか」をあなたの言語で丁寧に教えてくれます。

**AIとの協調開発で知っておくべきこと:**

**Q: コース形状をAIは知っているの？**
知りません。AIはコースの3D形状や見た目を直接知る手段を持っていません。ただし、このレッスンのルール・制約・入出力仕様（`05_Rule_Based_Control.md` など）や `rule_based_algorithms/` のコードを読ませることで、「どんな環境で動くか」をある程度理解させることはできます。「このファイルを読んで、コースの制約を踏まえた上で提案して」と明示的に指示するのがコツです。

**Q: AIが嘘をついたらどう見破る？**
AIが存在しない関数を書いた場合、実行すると `ImportError` や `AttributeError` が出ます。エラーが出たらコードの問題なので、**そのエラーメッセージをそのままAIに貼り付けて**「このエラーが出た、原因と修正方法を教えて」と聞きましょう。コードを走らせる前にAIに「このコードで使っている関数やモジュールはすべて実在しますか？」と確認させるのも有効です。

**Q: うまく動かない。戦略が悪いのかコードが悪いのか？**
切り分けの手順：
1. `print()` でアルゴリズムの中間値（`lateral_px`、`theta_deg` など）を確認 → 想定通りの値が出ているか？
2. デバッグ画像で白線が正しく検出されているか確認
3. 問題のある変数・挙動を特定してからAIに「この値がおかしい、なぜ？」と聞く

「なんか動かない」ではなく「どこが期待と違うか」を特定してからAIに聞くと、回答の質が大幅に上がります。

**実装アイデアへのQ&A:**

> **❓ 「sliding_windows.pyをOpenCV（Canny + HoughLinesP）に置き換えて高速化できる？」**
> できます。`opencv-python`は`requirements.txt`で既にインストール済みです。トレードオフを把握した上で選択しましょう。
> - **スライディングウィンドウ（現サンプル）**: カーブ追従が得意。実装が明快でデバッグしやすい。
> - **Canny + HoughLinesP**: 直線検出が高速。ただし曲線コーナーでは直線仮定が崩れる場合がある。
>
> 現サンプルはNumPy配列ベースの処理を使っており、Pythonのforループによるピクセル逐次処理ではありません。まず`SAVE_DEBUG_OVERLAYS=True`でデバッグ画像を確認し、実際の検出精度を確かめてから最適化手法を選ぶのが効率的です。

> **❓ 「driver_model.pyをPIDにアップグレードできる？現在はP制御だけ？」**
> その通り、P制御です。コードを確認すると `steer = k_theta * theta_rad + k_lateral * lateral_n` がまさに2入力の比例制御です。なお`alpha_smooth`は誤差のD項ではなく**出力のIIR平滑化**（`steer_s = (1-a)*steer + a*prev_steer`）です。
> PIDの追加：I項は`lateral_n`の累積和（**インテグレータワインドアップ対策が必須**）、D項は`lateral_n - prev_lateral_n`です。AIアシスタント（Gemini Code Assist、Claude Code、Codexなど）に「driver_model.pyのDriverModelクラスにI項とD項を追加してPIDにしてください。ワインドアップ対策も含めて」と依頼してみましょう。

> **❓ 「白線を見失ったときにEMAやカルマンフィルタで補間できる？」**
> 実はサンプルに**既に基本的な仕組みがあります**。`rule_based_input.py`の`_lost_age`カウンタと`HOLD_FRAMES`（2秒分）がそれです。白線を見失うと`hold`モード（最後の操作値を保持）→ 2秒後に`search`モード（サーチ操舵）に自動で切り替わります。EMAはこの`hold`ロジックの延長として、前フレームの検出位置を加重平均で補間する形で追加できます。カルマンフィルタはより高度なアプローチで、観測ノイズと予測ノイズを分離して激しく揺れる検出結果を平滑化するのに有効です。

## 5. アルゴリズム開発：勝つための戦略を立てよう
> **▶ 動画チャプター**: [アルゴリズム開発サイクルを体験する](https://youtu.be/kwLAPdCD-jU?t=464)

### 5.1 戦略を考える

あなたは今、**戦うべきフィールドにAI Agentとともに立っています。**

まずAIアシスタントと対話しながら、**方針（戦略）** を決めましょう。

> **💡 「CV最適化とPID、どちらから手をつけるべき？」という方へ**: まず**Section 6の課題4（評価メトリクス）を先に作ることを強く推奨します**。ベースライン計測なしにアルゴリズムを変えても「本当に改善したのか」を判断できません。スコアを記録してから改善に進むのが堅実なエンジニアリングです。

```
このルールベース制御システムを使って、
2周を最短タイムでゴールするための戦略を提案してください。
勝ち条件と制約条件を踏まえて、
現在のサンプルアルゴリズムのどこを改善すれば良いか教えてください。
```

### 5.2 開発の流れ

1. **方針決定**: AIアシスタントと相談して改善方針を決める
2. **実装**: `rule_based_algorithms/` 内のファイルをAIアシスタントと一緒に書き換える
3. **テスト走行**: `start.bat` を起動して実際に走らせる
4. **分析**: ターミナルログ・デバッグ画像・`metadata.csv` で結果を確認
5. **反復**: 改善してまた走らせる

あくまでサンプルプログラムなので、それを活用してもいいですが、逆にその思想にとらわれすぎないようにしましょう。AIアシスタントをうまく活用し、導くのがエンジニアの役目の一つです。

> **コツ**: サンプルのまま使ってもOKです。パラメーターを少し変えるだけでも走りが変わります。大きく書き換えるか、細かく調整するか、方針はあなた次第です。

### 課題4: アルゴリズムを定量評価するスクリプトを作る（上級）

「さっきよりスムーズになった気がする」という主観評価から卒業しましょう。`metadata.csv`から以下のメトリクスを自動計算するスクリプトをAIアシスタントと一緒に作ってください。

```python
# ステアリングのジッタ（滑らかさの指標 — 小さいほど滑らか）
jitter = df["steer_angle"].diff().dropna().std()
# 衝突発生率（0.0〜1.0 — 小さいほど良い）
collision_rate = (df["collision_type"] != "").mean()
# SOC消費効率（最終SOC / 走行時間 — 大きいほど省エネ）
soc_efficiency = df["soc"].iloc[-1] / (df["race_time_ms"].iloc[-1] / 1000)
```
